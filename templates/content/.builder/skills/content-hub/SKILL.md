# Content Hub Command

Plan and scaffold a content hub: a pillar page surrounded by 8-12 cluster pages. The hub command handles planning and scaffolding only. Per-page creation is manual via `/content-research`, `/content-write`, `/content-optimize`, `/content-polish`.

## Three Modes

| Mode               | Trigger                            | What It Does                                                 |
| ------------------ | ---------------------------------- | ------------------------------------------------------------ |
| **Plan** (default) | `/content-hub "topic"`             | Research, plan, scaffold folders with hub-context.yaml       |
| **Expand**         | `/content-hub --expand hub-slug`   | Add clusters to existing hub, scaffold new folders           |
| **Finalize**       | `/content-hub --finalize hub-slug` | Cross-cluster links, validation, pillar re-check, hub status |

## Arguments

<args> $ARGUMENTS </args>

```
IF arguments start with "--expand":  → Expand mode. Next token is hub_slug.
IF arguments start with "--finalize": → Finalize mode. Next token is hub_slug.
ELSE:                                → Plan mode. Entire string is the topic.
```

**If no arguments:** Ask the user: "What do you want to do? Provide a topic for a new hub, or use `--expand <slug>` / `--finalize <slug>` for an existing hub."

Do not proceed until you have valid arguments.

## Plan Mode (Default)

Plan a new hub and scaffold page folders.

### Pre-Flight: Ahrefs Budget Check

Before starting, check Ahrefs API units:

1. Call `subscription-info-limits-and-usage` via Ahrefs MCP
2. Hub planning uses ~3,000 units. The full hub (all pages via separate commands) may use up to 40,000 units total.
3. **If remaining < 40,000:** Warn: "Ahrefs budget is low ([X] units). Planning alone costs ~3,000 units. Continue with planning?"
4. **If Ahrefs MCP unavailable:** Stop. "Hub planning requires Ahrefs for cluster discovery. Cannot proceed."

Hub planning requires Ahrefs -- no WebSearch fallback for cluster discovery.

### Phase 0: Hub Planning

**Agent:** hub-strategist

#### Step 0.1: User Ideas Prompt

Before invoking the hub-strategist agent, ask the user:

"Do you have any cluster page ideas in mind? Share topics, angles, or subtopics you've been thinking about. Or say 'discover them' to let Ahrefs research find clusters."

Collect user ideas (if any) and pass as `user_cluster_ideas` to the hub-strategist agent.

#### Step 0.2: Hub Planning

Invoke the hub-strategist agent with:

- The user's broad topic
- `user_cluster_ideas` (if provided)

The agent runs the full hub planning process (validation, cluster discovery with user idea merge, scoring, link planning, publishing schedule).

**Output:** `hub.yaml` written to `output/hubs/<hub-slug>/`

### GATE 1: Hub Plan Approval

Present the hub plan using **AskUserQuestion**:

**Show the user:** Hub overview, pillar page, cluster table (label each as "Your idea" / "Discovered" / "Merged"), link plan, publishing schedule, Ahrefs budget estimate.

**File to review:** Tell the user: "Review `hub.yaml` in the output folder for the complete hub blueprint."

**Question:** "Hub plan ready. How do you want to proceed?"

**Options:**

1. **Approve** -- Scaffold page folders
2. **Modify clusters** -- Add, remove, or reorder cluster pages (re-runs relevant planning steps, re-presents Gate 1)
3. **Change priorities** -- Adjust creation order (updates hub.yaml, re-presents Gate 1)
4. **Stop** -- Abandon hub planning

**If Modify or Change Priorities:** Update `hub.yaml` with the changes. Re-present Gate 1 with updated plan.

### Folder Scaffolding (After Gate 1 Approval)

After the user approves the hub plan:

#### Step S.1: Create Directory Structure

```
output/hubs/<hub-slug>/
├── hub.yaml                         # Already created in Phase 0
├── pillar/
│   └── phases/
├── clusters/
│   ├── <cluster-slug-1>/
│   │   └── phases/
│   ├── <cluster-slug-2>/
│   │   └── phases/
│   └── ...
```

Create all directories using `mkdir -p`.

#### Step S.2: Write hub-context.yaml Files

Write a `hub-context.yaml` file into every page folder (pillar + each cluster).

**hub-context.yaml schema:**

```yaml
# Auto-generated by /content-hub. Read by per-page commands.
schema_version: 1
hub_slug: <hub-slug>
page_type: pillar | cluster
page_slug: pillar | <cluster-slug>

# --- Pipeline fields (consumed by commands to pre-populate Phase 1) ---
topic: "Page Topic Title"
primary_keyword: "page primary keyword"
content_goal: acquisition # From hub.yaml (page override or hub default)

# --- Reference fields (for human readability, not consumed by pipeline) ---
target_word_count: 2200
search_intent: commercial
business_potential: 3
priority: 1
```

**Hub.yaml path resolution:** Commands resolve hub.yaml deterministically as `output/hubs/<hub_slug>/hub.yaml`. No relative paths -- no `hub_yaml_path` field.

**Pipeline fields vs reference fields:**

- Pipeline fields (`hub_slug` through `content_goal`) are consumed by per-page commands to pre-populate Phase 1
- Reference fields (`target_word_count` through `priority`) are for human readability only -- not consumed by the pipeline

#### Step S.3: Update Hub Status

Set hub status to `scaffolded` in `hub.yaml`:

```yaml
status: scaffolded # planning → scaffolded → in-progress → published/partial
```

#### Step S.4: Display Result

Display the scaffolded folder tree:

```
Hub scaffolded!

Hub: [hub_name] ([hub-slug])
Content Pillar: [pillar]
Content Goal: [goal]
Clusters: [N]

output/hubs/<hub-slug>/
├── hub.yaml
├── pillar/
│   ├── hub-context.yaml
│   └── phases/
├── clusters/
│   ├── <cluster-slug-1>/
│   │   ├── hub-context.yaml
│   │   └── phases/
│   └── ...

Next steps:
1. Pick a page and run: /content-seed <folder-path>   (optional: add your own research)
2. Start the pipeline: /content-research <folder-path>
3. Write the post:     /content-write <folder-path>
4. Optimize:           /content-optimize <folder-path>
5. Polish:             /content-polish <folder-path>

Recommended order: Start with the pillar page, then clusters by priority.
Pillar: output/hubs/<hub-slug>/pillar/
First cluster: output/hubs/<hub-slug>/clusters/<highest-priority-slug>/
```

## Expand Mode (`--expand <hub-slug>`)

Add clusters to an existing hub.

### Step E.1: Load Existing Hub

Read `output/hubs/<hub-slug>/hub.yaml`. If not found, stop with error: "Hub '<hub-slug>' not found."

Display current hub status: hub name, status, cluster count, existing clusters with statuses, Ahrefs units consumed.

### Step E.2: Collect New Cluster Ideas

Ask the user: "What cluster topics do you want to add?"

### Step E.3: Validate New Clusters

For each new cluster idea:

1. Run `keywords-explorer-overview` -- get volume, KD, traffic potential
2. Run `serp-overview` -- check cannibalization against existing clusters and pillar
3. Score business potential (0-3)
4. Flag any issues (zero volume, cannibalization, low BP)

**Validation rigor:** New clusters undergo the same validation pipeline as Ahrefs-discovered clusters (hub-planning Steps 2B/2C). No free pass for user ideas.

### Step E.4: Present Updated Plan

Show new clusters alongside existing ones. Ask for approval:

**Options:**

1. **Approve** -- Add clusters and scaffold new folders
2. **Modify** -- Adjust the new clusters
3. **Stop** -- Cancel expansion

### Step E.5: Update Hub

After approval:

1. Append new clusters to `hub.yaml` with `status: planned`
2. Plan links for new clusters (pillar ↔ new cluster, new cluster ↔ existing clusters)
3. Scaffold new page folders with `hub-context.yaml`
4. **Regenerate hub-context.yaml for ALL existing pages** if hub-level `content_goal` or priorities changed
5. Present updated folder tree

## Finalize Mode (`--finalize <hub-slug>`)

Complete a hub by injecting cross-cluster links, validating all links, and updating hub status.

### Pre-Flight

Read `output/hubs/<hub-slug>/hub.yaml`. If not found, stop with error.

Check page statuses. If pages remain at `planned` or `in-progress` status, warn:

"N pages have not reached published status: [list]. Finalize with current pages or return to complete them?"

**Options:**

1. **Proceed** -- Finalize with published pages
2. **Stop** -- Return to complete remaining pages

### Finalization Process

Follow the detailed process in the hub-linking skill's [hub-finalization.md](.builder/skills/hub-linking/references/hub-finalization.md):

1. **Cluster-to-cluster link injection** -- Insert deferred sibling links into published cluster post.md files
2. **Final link validation sweep** -- Verify all links exist in post.md files, detect orphaned pages
3. **Pillar page re-validation** -- Re-run post-publish-checklist on pillar (modified since initial publish by reverse link patches)
4. **Hub status finalization** -- Set hub status (`published` / `partial`), record link counts
5. **Post-finalization recommendations** -- Style audit, compound docs, next steps

### Hub Summary

After finalization, display:

```
Hub finalized!

Hub: [hub_name] ([hub-slug])
Status: [published / partial]

Pages:
  Pillar: [title] ([word count] words) -- [status]
  Clusters: [published]/[total] published
    1. [cluster-title] -- [status]
    2. [cluster-title] -- [status]
    ...

Links:
  Total: [N] verified / [N] implemented / [N] planned / [N] errors
  Orphaned pages: [count]

Output: output/hubs/<hub-slug>/
```

### Next Steps

Use **AskUserQuestion**:

**Question:** "Hub finalization complete. What would you like to do next?"

**Options:**

1. **Capture learnings** -- Run `/content-compound` on this hub
2. **Style audit** -- Review all hub pages for voice consistency (recommended for hubs with 5+ pages)
3. **View hub** -- Open `hub.yaml` for the final blueprint
4. **Done** -- End

## Hub Status Transitions

```
planning → scaffolded → in-progress → published
                                     → partial
```

| Status        | Meaning                                             | Set By                                                   |
| ------------- | --------------------------------------------------- | -------------------------------------------------------- |
| `planning`    | Hub plan in progress                                | Hub-strategist agent (Phase 0)                           |
| `scaffolded`  | Folders created, no pages started yet               | Plan mode (Step S.3)                                     |
| `in-progress` | At least one page has started its pipeline          | Per-page commands (auto, when hub-context.yaml detected) |
| `published`   | All non-skipped pages published, links verified     | Finalize mode                                            |
| `partial`     | Some pages incomplete, user stopped or pages failed | Finalize mode                                            |

## Error Handling

### Ahrefs MCP Unavailable (Plan/Expand Mode)

Hub planning and expansion require Ahrefs for cluster discovery and validation. If unavailable, stop. Do not attempt WebSearch fallbacks.

### Hub Not Found (Expand/Finalize)

If `output/hubs/<hub-slug>/hub.yaml` does not exist: "Hub '<hub-slug>' not found. Check the slug or run `/content-hub \"topic\"` to create a new hub."

### Finalization Errors

If link injection fails for specific links, flag and continue. Do not block finalization on individual link failures. Present all errors in the final report.

## Known Gaps

- **Refresh commands** (`/content-refresh`, `/content-refresh-research`, `/content-refresh-write`) are not hub-aware in this version. Refreshing a hub page requires providing the full folder path manually.
- **No auto-scanning for write/optimize/polish:** These commands work with explicit hub page folder paths but do not auto-discover hub pages when run without arguments. The scaffold output prints paths for the user to copy.

## Important Notes

- The hub command creates folders and context files only. No page content is generated. Per-page creation happens through `/content-research`, `/content-write`, `/content-optimize`, `/content-polish`.
- `hub-context.yaml` is a write-once snapshot created at scaffold time. Per-page commands read mutable fields (status, content_goal) from `hub.yaml` via `hub_slug`.
- Hub-aware behavior in downstream skills (topic-discovery, outline-creation, seo-optimization, post-publish-checklist) activates when `hub_slug` is present in `phases/01-topic-validation.yaml`.
- Hubs are **evergreen only** for v1. If the user proposes a trending topic, recommend a standalone post instead.
- After hub finalization, strongly recommend a cross-hub style audit. Voice drift across 10+ pages is significant when pages are written in separate sessions.
- Context window safety: each page is created in its own session. The hub command does not carry forward page content between sessions.
