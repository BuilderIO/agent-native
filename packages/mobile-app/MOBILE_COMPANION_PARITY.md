# Mobile Companion Parity

This roadmap translates the strongest companion patterns from Wispr Flow, Granola, and Loom into Agent Native. It is intentionally capability-based: we copy useful jobs and system integrations, not another product's visual identity or proprietary implementation.

Status definitions:

- **Shipped in this slice** — implemented in the current mobile package and Clips action surface.
- **Next native milestone** — the next platform-specific targets required for competitive companion behavior.
- **Later** — valuable parity work that should follow the core capture and system-entry milestones.

## Cross-cutting release gates

Every capture surface must pass the same gates before release:

1. **Permission truthfulness:** the explanation shown before a system prompt matches the data actually accessed, and denial always has a non-destructive repair path.
2. **Visible recording:** microphone, camera, screen, accessibility, and call capture are never hidden; stop is reachable from the active system surface.
3. **Recovery:** once native capture has finalized a media file, app termination, interruption, expired auth, low storage, and network loss preserve either the media or an actionable failure state. Recovering a recorder-owned unfinished file after hard process death is gated on the platform-native journal/service milestone.
4. **Exactly-once delivery:** stable capture ids and acknowledged upload checkpoints prevent duplicate Clips artifacts.
5. **Private by default:** media stays in app-private storage, tokens remain app-scoped, lock-screen content is redacted, and local media is removed only after confirmed processing or explicit deletion.
6. **Framework ownership:** UI state is reflected through application state where applicable; server mutations use Clips actions/client methods rather than new ad hoc REST CRUD routes.
