# Custom design system support

Agent Native keeps the agent, actions, state, and framework-owned behavior
intact while allowing an application to own the visual implementation. The
contract is intentionally independent of Tailwind, Radix, Emotion, or any
other styling runtime.

## Architecture

The implementation has four cooperating layers:

1. **Semantic bridge.** `defineDesignSystem` registers the 17 component
   contracts and `ToolkitProvider` resolves them. Semantic props describe
   intent, emphasis, size, pending state, controlled values, open state, and
   change callbacks. `className` and `style` are optional escape hatches, not
   styling requirements.
2. **Headless feature controllers.** A controller owns state, actions,
   optimistic updates, access checks, and keyboard/interaction semantics. The
   default view and a customer view consume the same controller.
3. **Product-concept slots.** Surfaces such as `BuilderConnectCard` and
   `ChatHistoryRail` expose meaningful composition points without making
   customers recreate framework behavior.
4. **Ejection.** When configuration and slots are insufficient, eject the
   smallest owned unit. Ejection transfers presentation ownership only; it
   does not fork Core actions, persistence, auth, sharing access, or agent
   execution.

`defineTheme` is the token layer below the bridge. In v1 it is build-time only:
the same exported theme object is imported by `app/design-system.tsx` and the
Core Vite plugin, which emits the normalized CSS variables once. It accepts
light/dark palettes in standard CSS color formats, including OKLCH; dark
falls back token-by-token to light without auto-derivation.

## Delivery sequence

The epic is delivered as one continuous effort, with these review checkpoints:

1. Lock the semantic contract, default adapter, error boundaries, token
   normalization, precedence rules, public exports, and contract version.
2. Standardize all 22 templates on local UI adapters and `ToolkitProvider` so
   the seam remains load-bearing and drift-tested.
3. Rebase flagship Core surfaces onto shared primitives while preserving
   byte-identical default visuals: sharing, command menu, settings, secrets,
   team, and themed toaster.
4. Extract controller state from `BuilderConnectCard` and `ChatHistoryRail`;
   keep the default and custom render paths on one controller.
5. Extract settings and sharing controllers, including `ShareButton`, before
   advertising those surfaces as headless.
6. Ship the conformance kit, import-direction guards, codemods/tombstones,
   finer-grained eject units, and a second non-Tailwind fixture adapter.
7. Verify real integrations with the MUI and Ant Design Chat examples. Each
   example registers all 17 components, maps its own light/dark theme tokens,
   runs the conformance suite, and documents tokens-only boundaries such as
   assistant-ui, cmdk, and Tiptap internals.
8. Publish the docs, skills, recipes, and migration guidance only after the
   examples and linked-path guard are green.

## Invariants and compatibility

Behavior-heavy adapters own their overlay, portal, z-index, focus, and
Radix-interoperability decisions wholesale. The conformance kit checks those
behaviors, accessible roles, controlled state, keyboard interaction, selection,
focus restoration, and portal targeting. Each registered component is wrapped
in an error boundary so a broken customer adapter falls back to the default
adapter for that control only.

`designSystem.components.ActionButton` takes precedence over legacy
`components.Button`; the legacy API remains the lowest-precedence compatibility
path and emits a development warning when both effective APIs are supplied.
Adding components or optional props is a minor contract change. Required props,
removed props, or behavioral changes require a new major contract version and
are detected by the conformance kit.

The distribution point is `@agent-native/toolkit`, with explicit
`./design-system` and `./conformance` exports. Customer adapters are ordinary
npm packages imported by app-owned TypeScript modules; components are never
auto-detected or loaded from JSON. Examples are workspace projects, not public
templates or CLI catalog entries.
